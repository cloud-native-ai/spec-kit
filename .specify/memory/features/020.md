# Feature Detail: Qoder Support

**Feature ID**: 020  
**Name**: Qoder Support  
**Description**: Add Qoder as a supported CLI assistant across initialization, validation, documentation, and release distribution.  
**Status**: Implemented  
**Created**: 2026-03-29  
**Last Updated**: 2026-03-30

## Overview

Qoder Support enables Spec Kit users to choose Qoder as a first-class CLI-based assistant during project setup and ongoing project maintenance. The feature aligns initialization, dependency validation, refresh flows, user-facing documentation, and release outputs so teams can adopt Qoder without manual patching or inconsistent instructions.

## Latest Review

Generated implementation plan `006-add-qoder-support/plan.md` plus Phase 0/1 design artifacts, then decomposed the work into executable tasks in `006-add-qoder-support/tasks.md`. The review confirmed Qoder should be delivered as an officially approved assistant, identified the existing governance/documentation drift, and mapped the work into CLI, refresh, documentation, distribution, and audit task groups.

## Key Changes

1. Extended `src/specify_cli/__init__.py` assistant matrix with `qoder` metadata, CLI validation guidance, and `--ai` help updates.
2. Added Qoder command bootstrap generation to `.qoder/commands/` for initialization flows.
3. Updated governance/whitelist surfaces to include Qoder: constitution, plan template, agents prompt/template, installation docs, README, usage guide, and instructions content.
4. Extended instruction refresh compatibility to generate `QODER.md` symlink plus `.qoder/project_rules.md` handling.
5. Added contract, integration, and unit test coverage for Qoder init/refresh/support-surface/distribution rules.
6. Kept Feature 020 status as Implemented and synchronized task tracking with completed implementation work.

## Implementation Notes

- Treat Qoder as a CLI-based assistant with the same lifecycle expectations users already have for other supported CLI assistants.
- Keep Qoder naming, acquisition guidance, and support status consistent across product surfaces.
- Validate both new-project and existing-directory flows because users may adopt Qoder in either path.
- Preserve parity across supported script variants in distributed outputs.
- Upstream reference patches under `tmp/patches/qoder-main/` may inform implementation, but acceptance should be governed by the current specification and project governance.
- Shared governance and template assets used by other approved assistants are in scope and must be updated together to avoid drift.

## Future Evolution Suggestions

- Add automated release audits that compare assistant support claims across docs, help text, and packaged outputs.
- Track assistant adoption and support incidents to refine onboarding guidance for newly added assistants.
- Consider a more centralized assistant capability matrix to reduce future drift when new assistants are introduced.
